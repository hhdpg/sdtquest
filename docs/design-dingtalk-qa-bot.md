## 一、项目背景与目标

### 1.1 背景

企业内部业务系统日常使用中，用户频繁遇到操作指南、流程咨询、异常排查等问题，目前依赖人工答疑，效率低且知识难以沉淀。本项目旨在构建一个基于本地大模型的智能问答机器人，对接钉钉群聊，实现自动化答疑。

### 1.2 核心目标

1. **钉钉群智能问答**: 用户在钉钉群中 @机器人 提问，机器人自动调用本地大模型生成回答
2. **知识自动化构建**: 无现成文档，通过解析前端代码自动提取页面菜单、按钮操作、业务流程等知识
3. **问题自动归纳**: 对群内提问自动分类、统计、汇总，帮助管理员了解高频问题和知识盲区
4. **全本地化部署**: 模型推理和数据处理均在本地完成，数据不出企业网络

### 1.3 问题类型分布

| 类型 | 占比 | 示例 |
|------|------|------|
| 操作指南 | ~50% | "怎么创建订单？" "在哪里导出报表？" |
| 流程咨询 | ~25% | "审批流程是怎样的？" "提交后经过哪些步骤？" |
| 异常排查 | ~15% | "为什么提示权限不足？" "页面加载失败怎么办？" |
| 其他/闲聊 | ~10% | "你好" "谢谢" |

## 二、模型选型

### 2.1 生成模型

| 属性 | 值 |
|------|-----|
| **模型** | `qwen3.5:35b-a3b-instruct-4bit` |
| **架构** | MoE (Mixture of Experts，混合专家模型) |
| **总参数量** | 35B |
| **激活参数量** | ~3B（每次推理仅激活部分专家） |
| **量化方式** | 4bit (Q4_K_M) |
| **推理内存占用** | ~18-20GB（全部专家需加载，但激活参数少，推理速度快） |
| **上下文窗口** | 8K-32K tokens（可配置） |
| **部署方式** | Ollama 本地部署 |

**选型理由**:
- **MoE 架构优势**: 总参数量 35B 保证了知识容量和推理质量，但每次推理仅激活约 3B 参数，推理速度接近 3B 模型，远快于同参数量的 Dense 模型
- **4bit 量化适配硬件**: Mac M5 Pro 24GB 统一内存可流畅运行，留出系统和其他组件的内存空间
- **中文能力**: Qwen 系列在中文理解和生成方面表现优秀
- **Instruct 微调**: 已针对指令跟随场景优化，适合问答任务
- **无需额外微调**: 基座模型能力强，结合 RAG 即可满足需求

### 2.2 向量模型 (Embedding)

| 属性 | 值 |
|------|-----|
| **模型** | `bge-m3` (BAAI General Embedding - M3) |
| **参数量** | ~568M |
| **内存占用** | ~2-2.5GB |
| **向量维度** | 1024 |
| **最大输入长度** | 8192 tokens |
| **语言支持** | 100+ 语言（中文、英文等） |
| **检索能力** | 同时支持稠密检索(Dense)、稀疏检索(Sparse/Lexical)、多向量检索(ColBERT) |

**选型理由**:
- **多粒度检索**: 一个模型同时支持语义匹配和关键词匹配，无需额外部署 BM25，简化架构
- **中文优化**: BGE 系列专为中文场景优化，在中文检索任务上表现优异
- **长文本支持**: 最大 8192 tokens 输入，适合较长的代码描述和文档片段
- **多语言**: 如系统中夹杂英文术语，也能良好处理
- **Ollama 兼容**: 可直接通过 Ollama 加载运行，统一管理

### 2.3 内存预算

| 组件 | 内存占用 |
|------|---------|
| qwen3.5:35b-a3b-instruct-4bit | ~18-20GB |
| bge-m3 | ~2-2.5GB |
| 系统及其他进程 | ~2-3GB |
| **合计** | **~22-25GB** |

> **注意**: 24GB 内存处于紧平衡状态。建议：
> 1. Embedding 模型按需加载（仅在索引构建或检索时加载，非长驻）
> 2. 或使用 Ollama 的自动模型卸载功能，空闲时自动释放不活跃模型
> 3. 如出现内存压力，可将生成模型降级为 `qwen3:14b`（~9GB）
> 4. 关闭不必要的后台应用

## 四、核心组件设计

### 4.1 DingTalk Bot Service

#### 4.1.1 接入方式

采用钉钉 **Stream 模式**（WebSocket 长连接），核心优势：

| 对比项 | Stream 模式 | Webhook 回调 |
|--------|------------|-------------|
| 公网 IP | 不需要 | 必须有 |
| 本地部署适配 | 非常适合 | 需要内网穿透 |
| 实时性 | 高 | 一般 |
| 部署复杂度 | 低 | 高 |

#### 4.1.2 核心职责

1. **消息接收**: 通过 Stream 监听群内 @机器人 的消息
2. **即时反馈**: 收到消息后立即回复 "🤔 正在为您查找答案..."，降低用户等待焦虑
3. **异步处理**: 将问题投入消息队列，异步调用 RAG + LLM 生成回答
4. **消息发送**: 生成完成后，通过钉钉 API 发送 Markdown 格式回答
5. **会话管理**: 维护最近 3-5 轮对话历史，支持追问场景
6. **限流保护**: 同一用户 60 秒内限流，防止重复提问

#### 4.1.3 消息处理流程

```
收到消息
  │
  ├─ 是否@了机器人？ ── 否 → 忽略
  │
  是
  │
  ├─ 去重检查（消息ID） ── 重复 → 忽略
  │
  新消息
  │
  ├─ 立即回复 "正在思考..."
  │
  ├─ 提取问题文本（去除@部分）
  │
  ├─ 投入消息队列
  │
  └─ 异步Worker:
       │
       ├─ 问题分类 + 意图识别
       ├─ RAG 检索
       ├─ LLM 生成回答
       ├─ 格式化 + 引用标注
       │
       ├─ 更新之前的"思考中..."消息为最终答案
       └─ 记录问答日志
```

### 4.2 知识库构建 — 代码解析流水线

#### 4.2.1 解析策略

由于没有现成文档，从前端代码中自动提取系统操作知识是核心创新点。

**解析目标**:

| 提取内容 | 来源文件 | 提取方法 |
|----------|---------|---------|
| 页面路由和路径 | `routes/` 配置文件 | AST 遍历路由定义 |
| 导航菜单结构 | `menu/` 配置文件 | AST 遍历菜单树 |
| 页面按钮和操作 | 页面组件中的 `<Button>` 等 | AST + 正则匹配 |
| 表单字段和含义 | `<Form>` `<Input>` `<Select>` 等 | AST 提取属性 |
| 页面文案和标签 | i18n 国际化文件 / 组件内文字 | 文本提取 |
| API 调用映射 | `api/` 或 `services/` 目录 | AST 提取函数调用 |

**解析工具**: Tree-sitter — 支持 JavaScript/TypeScript/JSX/TSX 的 AST 解析，性能好，多语言支持

#### 4.2.2 知识结构化

解析后的原始数据需转换为结构化的知识文档，每个知识点包含：

```yaml
知识条目结构:
  id: "page_order-management_btn_create-order"
  type: "button"                    # page / button / form / workflow
  title: "新建订单"
  description: "在订单管理页面中，点击右上角「新建订单」按钮，填写订单信息后提交"
  page: "订单管理"
  path: "/order-management"
  source_file: "src/pages/order-management/index.tsx"
  related_actions: ["提交表单", "调用 API /api/orders/create"]
  tags: ["订单", "创建", "操作"]
```

#### 4.2.3 LLM 辅助丰富

代码中的变量名和注释往往不够直白，利用 Qwen 模型批量为每个知识点生成通俗易懂的描述：

```
输入（代码片段）:
<Button type="primary" onClick={handleCreateOrder}>新建</Button>

输出（知识描述）:
在订单管理页面中，点击「新建」按钮可以创建新订单。
点击后会弹出表单，需要填写客户名称、订单金额等信息。
```

此步骤离线批量执行，不影响线上推理性能。

#### 4.2.4 分块策略

按知识粒度组织，而非简单按字符数切割：

1. **页面级**: 同一页面的所有按钮/操作合并为一个综合文档
2. **功能级**: 跨页面的完整流程（如审批流程）单独成文
3. **元数据**: 每块附带页面名、路径、类型等元数据，支持过滤检索

#### 4.2.5 手动文档补充

预留接口支持后期添加人工编写的文档（Word/PDF/Markdown），与代码解析知识统一入向量库。

### 4.3 RAG Pipeline

#### 4.3.1 bge-m3 多粒度检索

bge-m3 的核心优势是一个模型同时支持三种检索模式，无需额外部署 BM25：

```
用户问题: "怎么删除用户？"
         │
         ├─ Dense Retrieval (稠密检索)
         │   → 语义匹配: "用户管理页面的删除操作说明"
         │
         ├─ Sparse Retrieval (稀疏检索)
         │   → 关键词匹配: "delete user API 文档"
         │
         ├─ ColBERT Retrieval (多向量检索, 可选)
         │   → 细粒度词组匹配
         │
         └─ RRF 融合排序
             → 合并去重，取 Top-K 返回
```

#### 4.3.2 向量数据库

**选型: ChromaDB**

| 特性 | 说明 |
|------|------|
| 部署方式 | pip install，零部署 |
| 持久化 | 自动持久化到磁盘 |
| 元数据过滤 | 支持按 type/page/tags 过滤 |
| 混合检索 | 内置支持 |
| Python 原生 | API 简洁易用 |

#### 4.3.3 两类问题的差异化处理

| 维度 | 标准操作类 | 灵活推理类 |
|------|-----------|-----------|
| 检索策略 | 高阈值 (相似度 > 0.8) | 中等阈值 (相似度 > 0.6) |
| 检索数量 | Top 5 | Top 8 |
| temperature | 0.3 (稳定、精确) | 0.7 (灵活、创造性) |
| 回答风格 | 严格基于知识库，找不到就说不知道 | 结合知识库 + 模型推理 |
| 适用场景 | "怎么操作X？" "X在哪里？" | "X报错了怎么办？" "为什么X不行？" |

#### 4.3.4 Prompt 设计框架

```
=== System Prompt ===
你是企业业务系统的智能问答助手，负责解答用户关于系统操作、业务流程和异常排查的问题。
回答要求：
1. 用中文回答，条理清晰，分步骤说明
2. 如果涉及具体页面或按钮，明确指出位置（用「」标注）
3. 优先基于知识库内容回答，不确定时诚实告知
4. 回答末尾附上参考来源

=== 相关知识库内容 ===
【文档1】{retrieved_doc_1}
【文档2】{retrieved_doc_2}
...

=== 对话历史 ===
{conversation_history}

=== 当前问题 ===
用户: {question}

请给出回答:
```

### 4.4 问题分析与汇总

#### 4.4.1 问题分类

初始阶段使用规则 + 关键词匹配，积累数据后训练轻量分类器（SVM/TextCNN）：

| 类别 | 关键词特征 | 示例 |
|------|-----------|------|
| 操作指南 | "怎么" "如何" "在哪里" "步骤" | "怎么创建订单？" |
| 流程咨询 | "流程" "审批" "步骤" "先后" | "采购审批流程是怎样的？" |
| 异常排查 | "报错" "失败" "为什么" "不了" | "提交订单报错怎么办？" |
| 闲聊/其他 | "你好" "谢谢" | "谢谢" |

#### 4.4.2 数据存储

每条问答记录入库（SQLite）：

| 字段 | 说明 |
|------|------|
| question | 原始问题 |
| category | 分类 |
| answer | 生成的回答 |
| sender_id | 提问人 |
| confidence | 回答置信度 |
| is_answered | 是否成功回答 |
| timestamp | 时间戳 |

#### 4.4.3 定时汇总

每日凌晨自动生成汇总报告，推送到管理群或邮件：

- 各分类问题数量分布
- 高频问题 TOP 10
- 未能回答的问题列表（提示补充知识）
- 活跃用户统计

## 七、部署架构

### 7.1 开发环境

```
Mac M5 Pro (24GB)
├── Ollama (本地)
│   ├── qwen3.5:35b-a3b-instruct-4bit    # 生成模型
│   └── bge-m3                             # Embedding 模型 (按需加载)
│
├── Python Bot Service
│   ├── FastAPI + dingtalk-stream
│   ├── RAG Pipeline (LlamaIndex + ChromaDB)
│   └── Question Analyzer
│
└── SQLite (数据存储)
```

### 7.2 启动流程

```bash
# 1. 确保 Ollama 运行中
ollama serve

# 2. 拉取模型 (首次)
ollama pull qwen3.5:35b-a3b-instruct-4bit
ollama pull bge-m3

# 3. 构建知识库 (首次 / 代码更新时)
uv run scripts/parse_code.py --project-root /path/to/frontend
uv run scripts/build_knowledge.py

# 4. 启动机器人服务
uv run src/main.py
```

### 7.3 Ollama 配置建议

```bash
# 环境变量配置 (可选)
OLLAMA_NUM_PARALLEL=2          # 最大并发推理数
OLLAMA_MAX_LOADED_MODELS=1     # 同时加载模型数(24GB内存建议1)
OLLAMA_KEEP_ALIVE=10m          # 模型空闲保持时间
```

## 八、关于模型微调的结论

### 8.1 结论：现阶段不需要微调，纯 RAG 方案足够

| 分析维度 | 结论 |
|----------|------|
| 标准操作类 (~50%) | RAG 检索知识库直接回答，效果最佳 |
| 灵活推理类 (~50%) | Qwen3.5 MoE 基座模型推理能力足够强 |
| 训练数据 | 无历史问答对，缺乏微调数据 |
| 硬件限制 | 24GB 内存跑 35B 推理已紧张，微调不现实 |
| 知识库变化 | 系统会迭代，RAG 更新成本远低于微调 |
| 可解释性 | RAG 可追溯引用来源，微调模型是黑盒 |

### 8.2 未来优化路径

如果上线后效果不佳，按优先级尝试：

1. **Prompt 优化** (零成本): 调整 Prompt 模板，添加 Few-shot 示例
2. **检索优化** (低成本): 改进分块策略、添加重排序
3. **知识库扩充**: 补充手动文档、添加操作截图描述
4. **轻量微调** (高成本): 积累 1000+ 问答对后，考虑 LoRA 微调 7B 模型

## 九、关键风险与缓解

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| **内存压力** (24GB 紧张) | 高 | Embedding 按需加载；备选降级至 14B 模型 |
| **推理延迟** (>30s) | 高 | 先回"思考中..."；MoE 激活参数少，实际推理较快 |
| **代码解析质量** | 中 | LLM 辅助丰富 + 人工审核修正 |
| **模型幻觉** | 中 | 低温度 + 强制引用知识库 + 找不到就告知 |
| **知识库外问题** | 低 | 相似度阈值 + 兜底话术 + 记录待补充 |
| **并发瓶颈** | 低 | 限流 + 队列控制并发 + 相似问题缓存 |

## 十、分阶段实施计划

### Phase 1: MVP 基础问答 (2-3 周)

**目标**: 跑通端到端问答流程

| 周次 | 任务 | 交付物 |
|------|------|--------|
| Week 1 | 钉钉开放平台接入 + Stream 模式消息收发 | 可收发消息的机器人 |
| Week 2 | Ollama 集成 + ChromaDB + 基础 RAG 管道 | 能基于知识库回答问题 |
| Week 3 | 端到端联调 + 手动知识库 + 内测 | 首版可运行的 MVP |

### Phase 2: 代码解析流水线 (3-4 周)

**目标**: 自动化从前端代码提取知识

| 周次 | 任务 | 交付物 |
|------|------|--------|
| Week 4-5 | Tree-sitter 解析器开发 (路由/菜单/按钮/表单) | 代码解析模块 |
| Week 6 | 知识结构化 + LLM 辅助丰富 + 向量化入库 | 自动化知识库 |
| Week 7 | 全链路测试 + 知识质量评估 | 知识构建 CLI 工具 |

### Phase 3: 问题分析汇总 (1-2 周)

**目标**: 问题分类、统计、汇总报告

| 周次 | 任务 | 交付物 |
|------|------|--------|
| Week 8 | 问题分类器 + 问答日志 + 汇总逻辑 | 每日汇总报告 |
| Week 9 | (可选) Streamlit Dashboard | 可视化面板 |

### Phase 4: 优化迭代 (持续)

- 检索质量优化（重排序、分块策略调整）
- Prompt 持续优化
- 功能扩展（图片识别、多轮追问等）
- 知识库增量更新机制
